# Part 4 Outline Authoring

你的任务是把 Part 3 已锁定的 argument tree 转译成 Part 5 可以直接写正文的中文论文大纲。大纲不是目录清单，也不是资料摘要；它是“论证路线到正文写作”的合约。

## 核心判断

每个章节都必须回答五个问题：

1. 读者到这一章需要理解什么？
2. 本章承接哪个 thesis、main argument、sub argument、counter 或 rebuttal？
3. 本章用哪些 research-wiki 页面与 `source_id` 支撑？
4. 本章和上一章、下一章的关系是什么？
5. 本章写完后，Part 5 作者能否据此展开正文？

无法回答这些问题的章节，不应进入 canonical outline。

## 输入边界

- 研究判断只能来自 `outputs/part3/argument_tree.json` 与 `research-wiki/`。
- `writing-policy/`、reference cases、rubrics 只提供结构、表达、篇幅和评审标准，不提供研究事实。
- 参考案例只能抽取抽象结构方法，例如“发展脉络 -> 类型分析 -> 机制分析 -> 案例验证 -> 结论回收”。不得迁移参考案例的主题、对象、人物、地点、案例名、历史事实、专有概念或研究结论。
- 若 reference case 与当前 argument tree 冲突，以当前 argument tree 为准，并把冲突写入 `outline_rationale.json` 或 `reference_alignment_report.json`。

## 章节职责

优先生成 5-7 个一级章节。每个一级章节只承担一个清晰的论证功能。

1. **绪论**：提出研究背景、问题意识、研究对象、范围、方法、材料来源和主论题。不要把绪论写成泛背景堆叠；它必须说明“为什么这个问题值得被研究”。
2. **研究现状与分析框架**：梳理已有研究、核心概念、方法基础和研究缺口。不要逐篇罗列文献；应说明当前论文将如何接过缺口并建立分析视角。
3. **对象界定或材料基础**：当 argument tree 需要先界定对象、类型、历史阶段、样本范围或评价标准时设置。本章的作用是建立后文论证的可比口径，不是重复绪论。
4. **核心论证章节**：每个 main argument 至少对应一个核心章节。章节标题应表达分析维度或论证动作，避免只用材料名、案例名或宽泛主题词。
5. **比较、案例或实证章节**：仅当 argument tree 要求通过比较、案例、调研、模拟、图表或类型分析来验证论点时设置。案例章节不能替代理论、方法或机制章节。
6. **策略、启示或转译章节**：仅当前文已经完成证据链，且 argument tree 明确包含实践指向时设置。策略必须从已证明的发现推出，不能新增研究事实。
7. **结论**：回收 thesis，概括主要发现、理论或实践贡献、局限和 Part 5 写作债务。结论不得提出未经前文章节论证的新观点。

## 结构模式选择

根据 argument tree 的主干选择一种骨架，不要机械套模板：

- **历史演进型**：问题提出 -> 阶段划分 -> 阶段机制 -> 当代问题或延续 -> 结论。
- **要素系统型**：研究框架 -> 关键要素分章 -> 要素关系综合 -> 代表性验证 -> 结论。
- **机制解释型**：现象界定 -> 影响因素 -> 作用机制 -> 证据验证 -> 方法或策略 -> 结论。
- **比较研究型**：比较对象与标准 -> 各对象分析 -> 差异机制 -> 综合判断 -> 结论。
- **案例研究型**：选例标准 -> 个案分析 -> 跨案例比较 -> 论点回收 -> 结论。
- **方法或评价型**：问题与指标 -> 方法建立 -> 数据或材料分析 -> 结果解释 -> 优化建议 -> 结论。

若 argument tree 同时包含多种逻辑，优先保留最能支撑 thesis 的一条主线，其余逻辑放入二级章节或 rationale 风险项。

## 写法规则

### 标题

标题应体现“研究动作 + 分析对象 + 论证方向”。优先使用：

- `<对象> 的形成机制`
- `<对象> 的类型划分与差异`
- `<问题> 的证据链与解释`
- `<方法> 下的 <对象> 分析`
- `<发现> 对 <实践目标> 的启示`

避免：

- “相关研究”“案例分析”“问题研究”这类无法看出论证职责的标题。
- 把参考案例中的具体主题词、地点、人名、项目名或结论带入当前大纲。
- 只按资料来源分类，而不是按论证任务分类。

### Brief

`brief` 用 1-2 句写清楚本节“要证明什么”和“怎么证明”，不要写成章节介绍。推荐句式：

```text
本章承担 <argument_node_id> 的论证任务，说明 <核心判断>。
本章将基于 <wiki/source 证据类型> 展开 <比较/归纳/机制解释/案例验证>，为下一章的 <承接任务> 提供前提。
```

### 二级章节

二级章节应拆分为“概念界定、材料范围、分析维度、证据组、反论点处理、阶段小结”等具体写作任务。不要把二级章节写成一级标题的同义重复。

## Schema 映射

写入 `paper_outline.json` 时：

- `section_id` 保持稳定、层级清晰，如 `sec_1`、`sec_1_1`。
- `title` 写用户可读的中文论文标题，不写内部节点名。
- `brief` 写章节论证任务，不写空泛说明。
- `argument_node_ids` 必须来自 canonical argument tree。
- `support_source_ids` 必须来自 argument tree 或 `raw-library/metadata.json` 可回溯来源。
- `writing_constraints` 只记录写作规范、表达限制、结构约束，不记录研究事实。

写入 `outline_rationale.json` 时：

- 说明为什么采用当前章节顺序。
- 记录每个章节承接的 argument nodes。
- 记录证据不足、wiki 回溯不足、reference case 不适配或 Part 5 需要谨慎处理的风险。

写入 `reference_alignment_report.json` 时：

- 只写 reference cases / rubrics 对“章节顺序、结构密度、论证展开方式、表达规范”的影响。
- 不写参考案例的主题内容。
- 若只使用了抽象模式，明确标注为 `structure_reference_only`。

## 质量自检

生成前检查：

- 只读一级标题，能否看出 thesis 的推进路线？
- 每个 main argument 是否至少被一个章节承接？
- 是否存在没有 argument_node_ids 的研究性章节？
- 是否存在有标题但无法写出正文的章节？
- 是否把文献综述写成文献清单，而不是研究缺口与分析框架？
- 是否把案例章节当成全文主体，却没有解释它支撑哪个论点？
- 是否在结论或策略章节新增了未经证明的判断？
- 是否混入了 reference case 的主题内容？

任一项失败，应先调整大纲，不要推进 Part 4 completion gate。
